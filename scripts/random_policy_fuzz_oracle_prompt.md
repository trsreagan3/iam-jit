# Random Policy Fuzz — Oracle Judgment Prompt

Copy-paste-ready prompt for an Opus-class Claude session to judge
random-policy-fuzz composites generated by
`scripts/random_policy_fuzz.py`. Self-contained; no other context
required.

This prompt is the **oracle phase** of the random-policy-fuzz
methodology (see `docs/RANDOM-FUZZ-METHODOLOGY-2026-05-22.md`).
The deterministic scorer's score is shown — the oracle MUST score
independently and not anchor on it.

---

## Prompt to paste into the Claude session

You are scoring AWS IAM policies for risk on a 1-10 scale. For
each composite below:

- **opus_score**: integer 1-10 (your independent risk judgment)
- **opus_factors**: list of 2-8 short risk-factor identifiers,
  using the canonical vocabulary below
- **opus_reasoning**: 1-3 sentences explaining the score

### Scoring rubric

| Score | Meaning |
|---|---|
| 1-2 | Effectively-zero impact: pure metadata reads, fully-scoped resource, or an explicit Deny dominates every Allow |
| 3 | Narrowly-scoped reads: single-service read on bounded resources |
| 4 | Broader reads or narrowly-scoped writes: read on Resource:* OR write on bounded resources |
| 5 | Write activity on a single service with no global blast-radius primitives |
| 6 | Multi-service writes; one or two Resource:* writes for non-sensitive services |
| 7 | Writes on sensitive services (IAM/KMS/Lambda/STS/S3-data) with some scoping |
| 8 | Broad writes on sensitive services; possible privilege-escalation primitive present but not composed |
| 9 | Privilege-escalation composition (e.g. iam:PassRole + lambda:CreateFunction on Resource:*); full-blast-radius destructive |
| 10 | Effective `*:*` admin or explicit account-takeover composition |

### Canonical risk-factor vocabulary

Use these identifiers (you may add free-form ones if needed):

- `admin_like` — effective wildcard on actions/services
- `public_principal` — trust policy allows external/unscoped principal
- `broad_resource` — Resource: `*` on a write-class action
- `priv_esc_passrole` — iam:PassRole composed with code-exec primitive
- `priv_esc_iam_management` — iam:Create/Update/Delete on Resource: `*`
- `code_execution_primitive` — lambda:CreateFunction / lambda:UpdateFunctionCode / cloudformation execute
- `destructive_action` — Delete*/Terminate*/Revoke* on Resource: `*`
- `secrets_read` — secretsmanager/SSM-SecureString/KMS-Decrypt on broad resource
- `sensitive_service` — KMS/IAM/CloudTrail/Organizations/CA touch
- `data_exfiltration_path` — broad s3:GetObject / dynamodb:Scan / rds:CreateDBSnapshot
- `effective_deny_dominates` — explicit Deny strictly contains every Allow → effective permission is near-zero
- `narrow_scope` — scoped to a single named ARN or bounded prefix
- `narrow_action_only` — actions are all read/list/describe only
- `narrow_condition` — Conditions meaningfully reduce blast radius
- `long_duration` — duration_hours ≥ 4 amplifies otherwise-acceptable shape

### Important — score independently

The composite includes iam-jit's deterministic score
(`scores.det_score`). **Ignore it when forming your judgment.**
Score the policy as if you had not seen the det_score. The
methodology's value is measuring the gap between your independent
score and iam-jit's; anchoring destroys signal.

### Output format

For each composite, output exactly one YAML block in this shape:

```yaml
- name: composite-NNNN-{seed}
  opus_score: <1-10>
  opus_factors:
    - <identifier>
    - <identifier>
  opus_reasoning: "1-3 sentences."
```

The founder will copy these YAML blocks back into each composite's
`scores.opus_*` fields, then run
`scripts/random_policy_fuzz_compare.py` to classify the gap.

### Inputs

Below this line, paste 1-N composite YAML files for judgment.
Each composite is the full YAML record from
`tests/calibration_corpus/random_composites/composite-NNNN-{seed}.yaml`.
You'll see the policy doc, the request context, and the
deterministic score with its factors. Judge independently.

## How the founder uses this prompt

1. Open a Claude (Opus) session.
2. Paste the section "Prompt to paste into the Claude session"
   above as the system / first user message.
3. Paste a batch of 5-15 composite YAMLs as the next message.
   Each composite is the full file content from
   `tests/calibration_corpus/random_composites/composite-NNNN-42.yaml`.
4. Collect Claude's YAML output blocks.
5. Merge each block back into the matching composite YAML's
   `scores.opus_*` fields (preserving `gap_classification:
   pending` — the comparison script flips that).
6. After ≥ N composites have full Opus scores, run:

   ```bash
   .venv/bin/python scripts/random_policy_fuzz_compare.py
   ```

   This produces `docs/RANDOM-FUZZ-RESULTS-{date}.md`.

## Why this prompt is structured this way

- **Independent scoring** is the whole point. The rubric +
  vocabulary are provided so Opus has a calibrated reference
  frame; the det_score is shown only for traceability.
- **Canonical factor vocabulary** keeps Opus's outputs
  comparable across sessions + over time. Free-form factors are
  allowed but discouraged.
- **YAML block output** lets the founder mechanically merge Opus
  output back into composite files without re-typing.
- **5-15 composites per batch** keeps each session focused; long
  batches risk Opus shifting its calibration mid-session.

## Constraints honored

- `[[scorer-is-ground-truth]]` — Opus judgment does NOT change
  the scorer; the comparison surfaces gaps for human review
- `[[calibration-quality-bar]]` — rubric + vocabulary are
  recorded here so future oracle sessions reuse the same scale
- No API calls from any iam-jit script — this is a manual
  Claude-session prompt
