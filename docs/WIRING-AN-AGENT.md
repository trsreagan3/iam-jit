# Wiring an agent through any bouncer

> **The Bounce suite is beta** — not yet recommended for production, and
> intentionally left out of the main README. It's defense-in-depth that gates an
> agent's *outbound* calls at runtime, alongside iam-jit's IAM scoping. This page
> is for people who want to experiment.

One page, four bouncers, one mental model: **start the bouncer, then point the
agent's client at it.** Each bouncer intercepts a different protocol, so the
"point the client at it" step differs — but the shape is always the same.

## Install the Go bouncers (beta)

`ibounce` ships with iam-jit (Python). The other three are separate Go binaries.
With a Go toolchain (Go ≥ 1.26; auto-fetched via `GOTOOLCHAIN` on Go 1.21+):

```bash
go install github.com/trsreagan3/kbouncer/cmd/kbounce@latest
go install github.com/trsreagan3/dbounce/cmd/dbounce@latest
go install github.com/trsreagan3/gbounce/cmd/gbounce@latest
```

Binaries land in `$GOPATH/bin` (defaults to `$HOME/go/bin`); add that to your
`PATH`. Homebrew tap + version pinning: [INSTALL-HOMEBREW.md](INSTALL-HOMEBREW.md).
These are the same paste-ready commands `iam-jit doctor install-check` prints.

If you only run one bouncer, you only need that one row. Most cloud-only setups
just need ibounce (the `AWS_ENDPOINT_URL` row).

> **Two wiring styles.** For ibounce + kbounce you can skip the manual env var
> entirely and let the CLI write the agent config:
> `ibounce mcp install-claude-code` (or `install-cursor` / `install-codex` /
> `install-devin`). See [MCP-RECIPES.md](MCP-RECIPES.md). The table below is the
> manual / transparent-proxy path, and the source of truth for the exact values.

## Per-protocol wiring table

| Bouncer  | Protocol it gates | What the agent sets | Exact value (default ports) |
|----------|-------------------|---------------------|------------------------------|
| **ibounce** | AWS API (SigV4) | `AWS_ENDPOINT_URL` | `http://127.0.0.1:8767` |
| **gbounce** | Generic HTTP/HTTPS | `HTTP_PROXY` + `HTTPS_PROXY` | `http://127.0.0.1:8080` |
| **kbounce** | Kubernetes API | `KUBECONFIG` → kbounce-routed config | file whose cluster `server: http://127.0.0.1:8766` |
| **dbounce** | SQL (PostgreSQL / MySQL) | client connection string / `PGHOST`+`PGPORT` | host `127.0.0.1`, port `5433` |

### ibounce — AWS SDK endpoint

```bash
export AWS_ENDPOINT_URL=http://127.0.0.1:8767
```

ibounce is a SigV4-preserving AWS-API proxy. Any AWS SDK (boto3, aws-sdk-js,
aws-cli) that honors `AWS_ENDPOINT_URL` routes through it with no other change.
This is the single env var the README leads with.

### gbounce — HTTP/HTTPS forward proxy

```bash
export HTTP_PROXY=http://127.0.0.1:8080
export HTTPS_PROXY=http://127.0.0.1:8080
```

gbounce is a standard forward proxy. Most HTTP clients honor `HTTP_PROXY` /
`HTTPS_PROXY` (lowercase variants too). For HTTPS interception you also need to
trust gbounce's CA — see [HTTPS-SETUP.md](HTTPS-SETUP.md).

### kbounce — Kubernetes via KUBECONFIG

```bash
export KUBECONFIG=~/.kube/kbounce-routed-config
```

kbounce gates the Kubernetes API. The agent's `kubectl` / client-go points at a
kbounce-generated kubeconfig whose cluster `server` is `http://127.0.0.1:8766`
(kbounce-generated configs carry a `# Generated by kbounce` marker, which
`iam-jit posture` scans for). The kubeconfig also carries the CA / TLS material
for the routed cluster; kbounce's own `init-tls` helper (in the `kbouncer`
repo) generates that material — run `kbounce --help` for the exact subcommand in
your installed version.

### dbounce — SQL connection string

```bash
# PostgreSQL example — point the client at dbounce's wire port:
export PGHOST=127.0.0.1
export PGPORT=5433
# or in a connection string:
#   postgresql://user:pass@127.0.0.1:5433/your-db
```

dbounce sits in front of PostgreSQL / MySQL on its **wire** port `5433`. Change
the client's host/port (or connection string) to `127.0.0.1:5433`; dbounce
forwards to the real database after gating each statement.

## Canonical port table

Single source of truth for every bouncer's ports. Each bouncer has a **wire /
proxy** port (where the agent's traffic goes) and a **management** port (health
+ audit). For ibounce and kbounce these are the same port; dbounce and gbounce
split them.

| Bouncer  | Wire / proxy port | Management port | Health probe |
|----------|-------------------|-----------------|--------------|
| **ibounce** | 8767 | 8767 (same) | `http://127.0.0.1:8767/healthz` |
| **kbounce** | 8766 | 8766 (same) | `http://127.0.0.1:8766/healthz` |
| **dbounce** | 5433 (SQL wire) | 8768 | `http://127.0.0.1:8768/healthz` |
| **gbounce** | 8080 (HTTP wire) | 8769 | `http://127.0.0.1:8769/healthz` |

These match the defaults pinned in the code
(`src/iam_jit/posture/bouncers.py`, `src/iam_jit/cli_uninstall.py`) and probed
by `iam-jit posture`. Every bouncer accepts a `--port` (and dbounce/gbounce a
`--mgmt-port`) flag to override.

> **Note on 8768 vs 8080.** A common doc-drift bug is showing gbounce's HTTP
> proxy as `8768` — that is **dbounce's management port**, not a proxy. The
> generic HTTP/HTTPS proxy (gbounce) is **8080**. If you see `HTTP_PROXY` on
> `8768` anywhere, it is wrong.

## Let the CLI emit the right exports

Instead of hand-setting env vars, start your bouncers and run:

```bash
eval "$(iam-jit shellinit)"
```

`iam-jit shellinit` probes which bouncers are actually running and emits only
the `export` lines for those — never a broken export for a stopped bouncer (per
[[ibounce-honest-positioning]]). It does **not** write to your shell rc files.

To verify the wiring after install:

```bash
iam-jit doctor install-check    # PATH, console scripts, MCP install support
iam-jit posture                 # which bouncers are running + how traffic is routed
```
