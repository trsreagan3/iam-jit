# `iam-jit posture` — what mode am I in?

**Launch-blocker §A42 / task #383.** This doc explains the unified
introspection surface that lets users *and* their agents answer
"am I behind iam-jit's scoped role / a bouncer / both / neither?"
in one command.

## The 4 modes

| Mode | iam-jit role | Bouncer intercepting | What protects you |
|---|---|---|---|
| **iam-jit + scoped role only** | yes | no | Ephemeral scoped IAM role narrows blast radius |
| **Bouncer only** | no | yes | Bouncer denies/audits at the proxy with ambient creds |
| **Both** | yes | yes | Belt + suspenders: scoped role + proxy gate |
| **Neither** | no | no | DIRECT — full ambient creds, no interception, no scoping |

Until #383 shipped, users + agents could **not** easily answer
which one of those they were in. Pieces existed (`/healthz`,
`bouncer_active_mode`, banners) but no unified surface.

## Surfaces

### Cross-product orchestrator

```bash
iam-jit posture                          # human-readable summary
iam-jit posture --json                   # structured for agents
iam-jit posture --check-direct           # warn loudly on DIRECT traffic
iam-jit posture --exit-1-on-unprotected  # CI gate use
```

Backed by `iam_jit.posture.capture_posture()` — also called by the
`iam_jit_posture` MCP tool so the agent sees identical data.

### Per-bouncer

| Product  | CLI                | MCP tool            |
|----------|--------------------|---------------------|
| ibounce  | `ibounce posture`  | `bouncer_posture` / `ibounce_posture` |
| kbounce  | `kbounce posture`  | `kbounce_posture`   |
| dbounce  | `dbounce posture`  | `dbounce_posture`   |
| gbounce  | `gbounce posture`  | `gbounce_posture`   |

Per `[[cross-product-agent-parity]]`: every Bounce ships the same
shape so an operator's muscle memory is uniform across protocols.

## Detection heuristics

### iam-jit-issued role

Three sources of evidence, in priority order:

1. `IAM_JIT_ASSUMED_ROLE_ARN` env var — set by `iam-jit assume` /
   `iam-jit request --print-creds-env`. **Authoritative if set.**
2. Role ARN path contains `/iam-jit/` (default CREATE-NOT-ASSUME prefix).
3. Local cache `~/.iam-jit/last-issued-role.json` matches the live
   ARN AND is still within its TTL.

When none of the above apply but an ARN is visible we report
`"unknown"`, **not** `False`. Per `[[ibounce-honest-positioning]]`:
no false negatives.

### Behind-a-bouncer

| Bouncer  | Env var checked                      | Process probe        |
|----------|--------------------------------------|----------------------|
| ibounce  | `AWS_ENDPOINT_URL` (loopback)        | TCP 127.0.0.1:8767   |
| kbounce  | `KUBECONFIG` (file w/ `# Generated by kbounce`) | TCP 127.0.0.1:8766 |
| dbounce  | `PGHOST` + `PGPORT` (loopback)       | TCP 127.0.0.1:8768 (mgmt) / 5433 (wire) |
| gbounce  | `HTTP_PROXY` / `HTTPS_PROXY` (loopback) | TCP 127.0.0.1:8769 (mgmt) / 8080 (wire) |

### Misconfiguration

Per `[[ibounce-honest-positioning]]`: if the env var points at a
loopback port but no bouncer is listening there, we **never**
silently claim "intercepted." We report:

```
MISCONFIG: AWS_ENDPOINT_URL=http://127.0.0.1:9999 but no bouncer
listening on that port
```

The SDK will fail-open or crash; either way the call is **not**
intercepted, and posture is honest about it.

## Privacy

- AWS role ARNs + env-var NAMES + loopback URLs pass through (they
  are identifiers, not secrets).
- Anything matching an AWS key pattern (`AKIA…`, `ASIA…`) is
  redacted as `[REDACTED]`.
- Any env var whose NAME ends in `_TOKEN` / `_SECRET` / `_KEY` /
  `_PASSWORD` is whole-value redacted regardless of shape.
- `aws_secret_access_key` values can **never** appear in the
  output — verified by `test_posture_does_not_leak_credentials`.

## Smoke tests

```bash
# State 1: nothing running
iam-jit posture
# -> ibounce / kbounce / dbounce / gbounce all STOPPED
#    all traffic classes DIRECT
#    tips: how to start each bouncer

# State 2: ibounce running in discovery mode
ibounce run --mode discovery &
AWS_ENDPOINT_URL=http://127.0.0.1:8767 iam-jit posture
# -> ibounce RUNNING; AWS intercepted; discovery-mode WARNING
#    (observes + audits but does NOT enforce denials)

# State 3: env var pointing at a down bouncer
AWS_ENDPOINT_URL=http://127.0.0.1:9999 iam-jit posture
# -> MISCONFIG line on stderr
#    AWS calls still DIRECT in effective_protection

# State 4: agent perspective
iam-jit posture --json | jq .overall_mode
iam-jit posture --json | jq .effective_protection.aws_calls
# Or via MCP from inside the agent runtime:
#   iam_jit_posture()  -> same dict
#   bouncer_posture()  -> just the ibounce block + identity
```

## CI gate usage

```yaml
# Fail the job if any traffic class is going DIRECT.
- name: Verify bouncer posture
  run: iam-jit posture --exit-1-on-unprotected
```

## Why this exists

Per the founder direction 2026-05-23:

> *"It should be easy for users/agents to know what mode they are
> using. For example, if they are using iam-jit with a scoped iam
> role, or are using bouncers, using both, or using neither."*

Without this, the marketing claim "iam-jit native" was
unsupported — an agent couldn't tell if it was *behind* iam-jit at
all. Now it can: every MCP-connected agent can call
`iam_jit_posture` before a sensitive op and condition its plan on
the answer.
