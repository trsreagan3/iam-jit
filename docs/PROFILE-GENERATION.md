# Profile generation — LLM-synthesized bounce profiles

**Task #326.** Generate bounce-suite profile YAML from either
observed audit events (primary) or a prose description of an
organization (secondary). Outputs are ALWAYS operator-reviewable
artifacts with the "STARTING POINT — review before distributing"
label baked in.

Per [[ibounce-honest-positioning]] the generator NEVER claims a
generated profile is "minimal," "PCI-compliant," "HIPAA-ready," or
similar. Compliance is the operator's call. Per
[[no-nl-synthesis]] this is distinct from IAM-policy synthesis
(which remains forbidden) — bounce profiles are layered config
artifacts on top of safety floors, not the floor itself.

## Two input paths

| Path | Headline use case | CLI | MCP tool |
|---|---|---|---|
| AUDIT-DRIVEN (primary, post-[[discovery-first-default]] pivot) | Operator runs a legitimate task in discovery mode, agent pins a profile from the last hour of usage | `iam-jit profile generate-from-audit` | `bounce_profile_generate_from_audit` |
| NL-CONTEXT DRIVEN (secondary) | Security team writes the org-base profile from scratch | `iam-jit profile generate` | `bounce_profile_generate` |

Both paths support `save` to persist the result under a named
bundle directory; per [[creates-never-mutates]] save refuses to
overwrite.

## Quick start — the killer UX

```bash
# 1. Operator runs a legit task; ibounce/kbounce/dbounce/gbounce are
#    in discovery mode (the post-pivot default), so everything passes
#    + everything is audit-logged.

# 2. Operator (or agent) asks: "what should the profile look like?"
iam-jit profile generate-from-audit \
    --time-range 1h \
    --add-safety-denies \
    --name "incident-response-runbook" \
    --output ./profiles/

# 3. Review the output bundle.
ls profiles/
# → index.yaml ibounce.yaml kbounce.yaml dbounce.yaml gbounce.yaml

cat profiles/ibounce.yaml | less
# → header: "STARTING POINT - based on 47 observed event(s) over 1h"
# → allows: scoped to exactly the observed resources
# → denies: the safety floor
# → flagged_for_review: broad globs the LLM emitted (none if tight)
# → skipped: events deliberately not turned into rules

# 4. Operator confirms the flagged_for_review list, then installs.
bounce profile install --from ./profiles/

# 5. Future runs use the pinned profile; prompt-injected attempts to
#    touch resources OUTSIDE the observed set get denied.
```

## Conversational agent usage (Claude saying "I'll pin a profile")

The MCP tools make this a natural agent surface. Sample agent
turn:

> **User:** I just finished the migration task. Pin a profile so
> next time we run anything in this account, we're only allowed
> to do what we did today.
>
> **Claude:** I'll pin a profile from the last hour of audit
> events. Let me query the bouncers first.
>
> `[calls iam-jit-audit-query MCP tool with time_range=1h,
>   agent_session=<current>]`
>
> Got 47 events. Now synthesizing the profile.
>
> `[calls bounce_profile_generate_from_audit MCP tool with
>   events=[...], add_safety_denies=true, name="post-migration"]`
>
> Generated. Two things to confirm:
>
>   - **Flagged:** the LLM emitted `arn:aws:s3:::reports-*-2026`
>     because all your reads were on the 2026 buckets. Is that
>     the right pattern, or do you want a specific bucket?
>   - **Skipped:** one call to `secretsmanager:GetSecretValue`
>     was a one-off — the LLM treated it as ambiguous (pattern
>     vs one-time). If it's a recurring need, I should add it
>     explicitly.
>
> Tell me how to resolve those and I'll re-generate, then save +
> install.

This is the conversational shape `[[deliberate-feature-completion]]`
expects: ASK before guessing on ambiguous patterns.

## Honest framing surfaces

Every generated artifact carries these visible markers:

### 1. STARTING-POINT label
At the top of every profile YAML:
```
# STARTING POINT - based on 47 observed event(s) over 1h; review
# before distributing.
# Generated by iam-jit profile generate
# (see docs/PROFILE-GENERATION.md).
# Per [[ibounce-honest-positioning]] this profile makes NO
# compliance claims; the operator's review IS the compliance step.
```

### 2. Provenance block
```yaml
provenance:
  source: 'llm-generated-from-audit'
  llm_backend: 'anthropic'
  audit_window_start: '2026-05-22T17:00:00Z'
  audit_window_end: '2026-05-22T18:00:00Z'
  source_session_id: 'abc-123'
  events_analyzed: 47
```

### 3. `flagged_for_review` list
Any pattern with a `*` inside a resource segment lands here for
operator confirmation. The generator auto-flags these client-side
even if the LLM didn't.

### 4. `skipped` list
Things the LLM deliberately did NOT include (with reasons):
- sparse one-off calls with ambiguous pattern intent
- events outside the requested scope

### 5. Budget tracking
The MCP tool returns `budget_spent_usd` (best-effort estimate from
the backend's `estimate_cost_per_1k`). Per
[[per-customer-llm-budget-cap]] hosted tier respects the per-
customer budget cap (#82); self-host uses the operator's own LLM
choice and pays nothing to iam-jit.

## CLI reference

### `iam-jit profile generate-from-audit`

```
iam-jit profile generate-from-audit \
    --agent-session abc-123 \
    --time-range 1h \
    --bouncer ibounce --bouncer kbounce \
    --add-safety-denies \
    --name "incident-response-runbook" \
    --output ./profiles/ \
    [--since ISO8601 --until ISO8601] \
    [--filter EXPR ...] \
    [--limit 500] \
    [--preferred-backend anthropic|openai|bedrock|ollama] \
    [--format json|yaml-bundle]
```

Behavior:
- Fans out to every reachable bouncer's `/audit/events` (the same
  endpoint `iam-jit audit query` uses) per
  [[cross-product-agent-parity]].
- Merges + sorts events, passes them to the LLM with a strict
  output schema.
- Layers the safety floor on top if `--add-safety-denies` (default).
- Writes a bundle directory matching the
  `docs/examples/profiles/index.yaml.template` shape.
- Refuses to overwrite an existing bundle dir per
  [[creates-never-mutates]].

### `iam-jit profile generate`

NL-context driven. Used by security teams writing the org-base
profile without observed traffic.

```
iam-jit profile generate \
    --context "Mid-size SaaS, prod/staging split, 5 engineers" \
    --start-from example-org-base \
    --name my-org-base \
    --output ./profiles/ \
    [--preferred-backend anthropic|openai|bedrock|ollama] \
    [--explain]
```

### `iam-jit profile save`

Wrap an already-edited YAML into a named bundle for install.

```
iam-jit profile save --as <name> path/to/profile.yaml
```

## Backend selection

Per [[pluggable-llm-backend-decision]] the generator respects the
standard backend-selection chain:

1. `--preferred-backend` flag / `preferred_backend` MCP arg
2. `IAM_JIT_LLM_BACKEND` env var
3. `IAM_JIT_LLM` env var (back-compat)
4. First-available in autoselect order: anthropic → openai →
   bedrock → ollama

When no backend is available the generator falls back to
**deterministic synthesis**:

- Audit-driven: every observed resource becomes an exact-match
  allow (no wildcards inferred); safety floor still layered on
  if `--add-safety-denies` set; `flagged_for_review` carries a
  "LLM unavailable" note so the operator knows.
- NL-context-driven: the safety floor across all four bouncers
  is emitted with no allows; `parser_strict_match=false`.

The deterministic fallback is intentionally less clever than the
LLM — it does NOT try to infer ARN globs from common prefixes.
Better to emit an event-literal profile the operator can broaden
than to silently widen scope.

## Bundle layout

```
./profiles/
├── index.yaml             # bundle metadata + sha256s; the file engineers install
├── ibounce.yaml           # AWS-specific allows + denies
├── kbounce.yaml           # K8s-specific
├── dbounce.yaml           # SQL-specific
└── gbounce.yaml           # HTTPS-specific
```

Install via the existing single-file or multi-file install path
(#200): `bounce profile install --from ./profiles/`.

## Safety floor (the `--add-safety-denies` content)

When `add_safety_denies=true` (default), the generator layers
these denies on top of the LLM's allows regardless of whether
the LLM emitted them:

| Bouncer | Targets | Reason |
|---|---|---|
| ibounce | `iam:CreateAccessKey`, `iam:CreateUser`, `iam:CreateLoginProfile`, `iam:UpdateLoginProfile`, `iam:Attach*Policy`, `iam:Put*Policy` | agents must not create credentials or escalate privileges |
| ibounce | `arn:aws:iam::*:role/break-glass-*` `sts:AssumeRole` | break-glass roles require human approval |
| ibounce | `kms:ScheduleKeyDeletion`, `kms:DisableKey`, `kms:DeleteAlias` | KMS deletion is irreversible |
| ibounce | `cloudtrail:StopLogging`/`DeleteTrail`/`UpdateTrail`, `config:Delete*Recorder`/`Stop*Recorder` | audit-infra destruction is admin-only |
| kbounce | cluster-scoped `delete`/`deletecollection` on `namespaces`/`nodes`/`clusterroles`/`clusterrolebindings` | cluster-destructive verbs require human approval |
| kbounce | cluster-scoped `get`/`list`/`watch` on `secrets` | cluster-wide secrets exfil |
| dbounce | `GRANT * TO PUBLIC`, `GRANT ALL PRIVILEGES TO PUBLIC` | silent privilege escalation |
| dbounce | `DROP SCHEMA pg_catalog*`, `ALTER SCHEMA pg_catalog*`, `DROP DATABASE mysql`, `DROP DATABASE information_schema` | system-catalog DDL |
| gbounce | `169.254.169.254` | IMDS access = credential exfil |
| gbounce | `dns.google` | DoH bypass of egress filtering |

These match the `docs/examples/profiles/example-org-base.yaml`
shipped at commit 6088cf1.
