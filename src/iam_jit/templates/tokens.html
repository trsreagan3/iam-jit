How agents create their own tokens (if you have a session, you can mint via API)

Once you've signed in via the magic link, your browser session can mint a token. An agent driving your browser can use this — or you can copy the token shown above into the agent's config.

Mint a new token

curl -X POST {{ public_url }}/api/v1/tokens \
  -H 'Authorization: Bearer <your-existing-token>' \
  -H 'Content-Type: application/json' \
  -d '{"label": "my-agent"}'

Or, when you're already logged in via cookie (e.g. an agent reusing the same browser session), you can include your iam_jit_session cookie:

curl -X POST {{ public_url }}/api/v1/tokens \
  -H 'Cookie: iam_jit_session=<your-session-cookie>' \
  -H 'Content-Type: application/json' \
  -d '{"label": "my-agent"}'

Use the token

curl {{ public_url }}/api/v1/users/me \
  -H 'Authorization: Bearer iamjit_...'

List / revoke your tokens

# list (returns hashes only, never raw values)
curl {{ public_url }}/api/v1/tokens -H 'Authorization: Bearer iamjit_...'

# revoke
curl -X DELETE {{ public_url }}/api/v1/tokens/<token_hash> \
  -H 'Authorization: Bearer iamjit_...'

Submit a request from an agent

For raw-JSON submission: POST /api/v1/requests with the bearer token. For agent-driven authoring with codebase context, run the iam-jit MCP server locally (iam-jit mcp-server) and use the list_templates / get_template / score_iam_policy / submit_policy tools — see docs/AGENTS.md.

Tokens never expire on their own. Revoke them when you're done. There's no rate limit yet — just don't share tokens between agents.
